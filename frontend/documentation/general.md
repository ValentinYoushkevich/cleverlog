BRD: Система учёта рабочего времени (Web) 

1. Введение 

Проблема. В компании нет единого инструмента учёта времени: каждый проект ведёт собственные Excel, HR вручную сводит по проектам и проверяет норму часов. Это неэффективно и ведёт к ошибкам, задержкам и неточному биллингу. 

Цели. 

Автоматизировать сбор логов рабочего времени. 

Обеспечить прозрачность трудозатрат по проектам. 

Контроль выполнения месячной нормы (часов) и выявление недо/переработки. 

Ускорить формирование отчётов для руководства (через сводный отчёт и экспорт в Excel). 

Предоставить админам дашборд для оперативного контроля. 

Фигура 

2. Ролевая модель и права 

Роли: 

User — вводит, редактирует, удаляет только свои рабочие логи и отсутствия; видит только свои данные (логи, календарь, отчёты). 

Admin — суперправа над всей системой, включая проекты, пользователей, календарь, норму часов, логи всех пользователей, закрытие месяца, кастомные поля. 

Права User (основное): 

Ввод/редактирование/удаление своих Work Logs и Absences (Vacation/Sick Leave/Day Off). 

Фильтры: проект, период, тип записи, поиск по комментариям, Task Number, кастомные поля (если активны для проекта). 

Просмотр своих отчётов и календаря (read-only при закрытом месяце). 

Экспорт своих логов в Excel (структура утверждена). 

Права Admin (полный доступ): 

Управление пользователями: создание, инвайт, изменение роли, перевод в Active/Inactive, редактирование профиля. 

Управление проектами: создание, изменение, статусы (Active/On Hold/Closed), обратимые переходы (в любую сторону). 

Конструктор кастомных полей для Work Logs (Text/Number/Dropdown/Checkbox), привязка к проектам, обязательность. 

Управление календарём: норма часов на месяц (в часах), настройка рабочих/выходных/праздничных дней. 

Полный CRUD над логами (включая любые операции в закрытом месяце). 

Закрытие/открытие месяца; просмотр и экспорт отчётов. 

Дашборд: диаграммы, списки по недо/переработке, незаполненным дням. 

Журнал аудита: все события (CRUD по ключевым сущностям, входы/неуспешные входы и т.д.). 

Фигура 

3. Основные сущности и словарь 

Сущности: 

User: Имя, Фамилия, Email (логин), Роль (User/Admin), Должность, Статус (Active/Inactive), Дата найма, Дата увольнения, Тэги. 

Project: Название, Статус (Active/On Hold/Closed), Перечень кастомных полей (включённых для проекта). 

Work Log: Пользователь, Дата, Проект, Длительность (ввод m/h/d, смешанный ввод допускается), Комментарий (обяз.), Task Number (обяз.), Кастомные поля (по проекту). 

Absence (Отсутствие): Тип (Vacation/Sick Leave/Day Off), Диапазон дат → создаются записи только по рабочим дням; длительность по умолчанию 1 d на день (редактируема), Комментарий — необязателен. 

Calendar: Рабочие/Выходные/Праздничные (настраивает Admin), Норма часов на месяц (например, 168 h). 

Custom Field: Название, Тип (Text/Number/Dropdown/Checkbox), Обязательность, Привязка к проектам. 

Audit Log: событие, актор, объект, before/after, timestamp, IP. 

Определения: 

Глобальная норма дня: 1 d = 8 h = 480 m (фиксировано). 

Факт месяца: Work + Absence (в часах). 

Норма месяца: задаёт Admin (в часах), общая для всех. 

Отклонение: Факт – Норма (используется для подсветки строк). 

Фигура 

4. Бизнесправила (свод) 

В будущее логировать нельзя (ни User, ни Admin). 

В прошедшие даты логировать можно без ограничений. 

User видит все проекты в системе (для логирования — только Active; On Hold/Closed скрыты в выборе проекта). Admin видит все. 

В On Hold/Closed логировать нельзя (для всех). 

В один день запрещено комбинировать Work Log и Absence. 

В день допускается несколько Work Logs (по разным проектам). 

Выходные/праздники (серые) не требуют логирования; Work Log можно; Absence — нельзя (и переносить на выходной — тоже нельзя). 

Диапазон Absence пропускает выходные автоматически; также пропускает дни, где есть Work Logs (и сообщает об этом). 

Календарная перекраска:  

зелёный — есть Work Logs в день, 

синий — Absence (на рабочем дне), 

красный — рабочий день без записей, или частично заполнен (менее 8 h) 

серый — выходной/праздник (можно логировать Work Log). 

Закрытие месяца: User — read-only; Admin — может всё. 

Кастомные поля: включаются perproject; отключение — скрывает поле в UI/отчётах/экспорте, но данные сохраняются (и вернутся при включении). Переименование — только отображаемое имя. Смена типа запрещена, если поле использовано. Dropdown: удалённые опции помечаются как «устаревшие» в старых логах. Удаление поля — soft delete (скрыть, данные оставить, можно восстановить). 

Фигура 

5. Ввод времени и отсутствий 

Ввод длительности: 

Допустим ввод: 15 m, 2 h, 1 d, комбинированный: 1 h 30 m, 1 d 2 h. 

Внутри система хранит в днях, UI и экспорт показывают часы. 

По умолчанию отображение в интерфейсе и в отчётах/экспорте — часы. 

Work Log (форма): 

Дата (один день, не диапазон). 

Проект (User — только Active). 

Длительность (m/h/d, смешанный ввод). 

Комментарий — обязателен. 

Task Number — обязателен. 

Кастомные поля проекта (по настройкам; обязательные/необязательные). 

Absence (Vacation/Sick/Day Off): 

Диапазон дат → создаются записи по рабочим дням внутри диапазона. 

Длительность по умолчанию = 1 d на день (изменяемо постфактум). 

Комментарий — необязателен. 

Все три типа засчитываются в норму. 

Absence в выходные — нельзя; при переносе на выходной — ошибка. 

При пересечении с существующим Work Log — соответствующие дни пропускаются с уведомлением.

User (только по своим данным): 

Проект (Active). 

Период: день/неделя/месяц/произвольный диапазон. 

Тип записи: Work / Vacation / Sick Leave / Day Off. 

Поиск по комментариям (подстрока). 

Поиск по Task Number (точное/маска). 

По кастомным полям (активным для выбранного проекта). 

Admin: 

Всё перечисленное + по пользователю (любому) + по проектам любым статусам в отчётах. 

Фигура 

7. Отчёты 

(1) Отчёт по пользователю (на экране): 

Колонки: Дата, Тип записи, Проект, Task Number, Длительность (ч), Комментарий, Кастомные поля. 

Итоги: Итого за период (часы); Разбивка по проектам; Разбивка по типам (Work/Vacation/Sick Leave/Day Off). 

Если период = полный месяц → показывать блок «Факт/Норма/Отклонение» (только на экране). 

Экспорт Excel — по согласованной структуре. 

(2) Отчёт по проекту (на экране): 

Включает только Work. 

Колонки: Пользователь, Дата, Проект, Должность, Task Number, Длительность (ч), Комментарий, Кастомные поля. 

Итоги: Итого по проекту (часы), Разбивка по пользователям (сумма часов). 

Фильтры: период, пользователь, Должность, Task Number, комментарий, кастомные поля. (Тип записи — не нужен.) 

(3) Свод по месяцу (таблица): 

Колонки: User, Project A, Project B, …, Absence (h) (сумма Vacation+Sick+Day Off), Факт (h). 

Норму и отклонение НЕ показываем колонками. 
Подсветка строки:  

зелёный — Факт == Норма, 

желтый — Факт ≠ Норма (недо/переработка). 

Итоговая строка TOTAL внизу по всем столбцам (суммы), без Нормы/Отклонения. 

(4) “Кто не логировал дни” — не Excelотчёт, а виджет+список на дашборде: 

Виджет: «🔴 X сотрудников не заполнили рабочие дни». 

По клику: список с ФИО, количеством незаполненных дней, датами, фактом за месяц, датой последнего лога, переходом в календарь. 

Экспорт в Excel (общая структура строк): 

Дата, Тип записи, Проект, Task Number, Длительность (в часах), Комментарий, Пользователь (для админа), Кастомные поля. 

Фигура 

8. Дашборд Admin 

Состав: 

Список всех проектов (фильтр по статусу). Клик → отчёт по проекту. 

Круговая диаграмма: суммарные часы по проектам за период. 

Круговая диаграмма: распределение сотрудников по проектам (%). 

Карточки:  

Кол-во людей с недоработкой (кликабельно → список). 

Кол-во людей с переработкой (кликабельно → список). 

Сотрудники с незаполненными днями (кликабельно → список). 

Списки «Переработка» и «Недоработка»:  

Пользователь, Факт (h), Absence (h), TOP2 проектов пользователя, кол-во незаполненных дней, дата последнего лога, переходы к календарю/отчётам. 

Все виджеты зависят от выбранного месяца/периода. 

Фигура 

9. Календарь (User/Admin) 

Цветовая логика: 

🟩 зелёный — есть Work Logs, 

🟦 синий — есть Absence Logs, 

🟥 красный — рабочий день без записей, 

⚪ серый — выходной/праздник. 

Поведение: 

Серые дни не требуют логирования, но Work Log можно; Absence — нельзя. 

Admin может менять статус дня (рабочий ↔ выходной/праздничный). 
Данные не удаляются; перекраска визуализации:  

выходной→рабочий + есть Work Log → день становится зелёным. 

Календарь учитывает норму месяца (для подсветок и итоговых блоков). 

По клику на день — список записей с действиями (добавить/редактировать; у User — read-only при закрытом месяце). 

Фигура 

10. Пользователи 

Поля: Имя, Фамилия, Email (логин), Роль (User/Admin), Должность, Статус (Active/Inactive), Дата найма, Дата увольнения, Тэги. 

Аутентификация: 

Admin создаёт пользователя (имя, фамилия, email, роль, и т.п.); система отправляет инвайт. 

Пользователь сам задаёт логин/пароль при регистрации. 

Только логин+пароль (SSO/2FA — нет в MVP). 

Статус Inactive (итоговое поведение): 

Не может входить; активные сессии инвалидируются. 

Admin может редактировать/создавать/удалять его логи задним числом. 

Не получает уведомления; скрыт в списках по умолчанию (включается фильтром). 

Фигура 

11. Проекты 

Поля: Название, Статус (Active/On Hold/Closed), список кастомных полей, применимых к проекту. 

Статусы: 

Active — логирование разрешено. 

On Hold — логирование запрещено (для всех); проект виден в отчётах/фильтрах. 

Closed — логирование запрещено; виден в отчётах. 

Admin может менять статусы в любую сторону (включая Closed → Active). 

Видимость в выборе проекта (при создании Work Log): 

User — только Active. 

Admin — видит все (логировать можно только в Active). 

Фигура 

12. Кастомные поля (конструктор) 

Типы (MVP): Text, Number, Dropdown (с набором опций), Checkbox. 
Привязка: включаются/выключаются perproject; настройка обязательности. 
Кто создает: Admin 
Бизнесправила: 

Отключение — поле скрывается (UI/отчёты/экспорт), данные сохраняются; при включении возвращаются. 

Переименование — меняется только отображаемое имя. 

Смена типа — запрещена, если поле уже использовалось. 

Dropdown: удалённые опции остаются в старых логах как «(устаревшее значение)», но не доступны для новых логов. 

Удаление поля — soft delete (скрыть; данные оставить; можно восстановить). 

Фигура 

13. Уведомления 

Пользователям: автоматическая рассылка в 09:00 последнего рабочего дня месяца (по календарю Admin), если:  

есть красные рабочие дни, 

или Факт < Норма, 

или отсутствуют логи за последний рабочий день. 

Админам письма не отправляются. 

Admin может отключить рассылку:  

глобально (для всех), 

точечно (для конкретного пользователя).

Попытки пользователя изменить данные в закрытом месяце блокируются («Месяц закрыт»).

16. Нефункциональные требования (NFR) 

Производительность и SLA: 

До 200 активных пользователей. 

Отклик UI ≤ 2 сек (типовые операции). 

Отчёты/дашборды ≤ 10 сек (1 месяц), ≤ 15 сек (до 3 мес). 

Экспорт Excel ≤ 15 сек (месяц), ≤ 30 сек (до 3 мес). 

Доступность (Availability) = 99.0% в месяц. 

Окно обслуживания: 00:00–06:00. 

Безопасность: 

Аутентификация: email + пароль (2FA/SSO — нет в MVP). 

Политика паролей: минимум 8 символов, 1 заглавная, 1 строчная, 1 цифра, 1 спецсимвол. 

Хранение паролей: Argon2id (при недоступности — bcrypt cost ≥ 12). 

HTTPS везде, HTTPonly cookies. 

Блокировка после 5 неверных попыток на 5–10 минут. 

Разделение ролей (RBAC). 

Хранение данных: 

Бессрочно (Work Logs, Absences, Projects, Users, Audit Log, Calendar history). 

Бэкапы/DR: 

Backup: ежедневно (full). 

RPO = 24 часа; RTO — не нормируется («как получится»). 

Логи и мониторинг: 

Backend technical logs: error + warning. 

Нет автоматического мониторинга, только серверные логи (без алёртов). 

Frontend JSошибки → отправляем на backend /log-js-error → пишем в technical log. 

Локализация и браузеры: 

Только русский (вся UI-строка в словарях). 

Desktopтолько (без адаптивной мобильной версии). 

Поддерживаемые браузеры: Chrome, Firefox, Edge, Safari Desktop (актуальные стабильные версии ±2). 

UX/UI принципы: 

SPAповедение (без перезагрузок страниц). 

Liveвалидация форм (ошибки сразу). 

Таблицы: сортировка, поиск, пагинация, sticky header. 

Обновления UI — сразу после изменений. 

Единый UIкит (общие элементы, единый стиль).

Пользователи: создание/инвайт/смена ролей/активация/деактивация/профиль. 

Проекты: CRUD/статусы/кастомные поля perproject. 

Календарь: изменение статуса дня, изменение нормы месяца. 

Закрытие/открытие месяца. 

Кастомные поля: CRUD, включение/отключение, переименование, изменение обязательности, dropdownопции, удаление (soft), восстановление. 

Входы/неуспешные входы/сброс пароля/смена пароля. 

Состав записи: 

timestamp, actor, role, event_type, entity_type, entity_id, before/after (JSON), ip, result. 

Доступ: только Admin (в UI). Фильтры, поиск, экспорт CSV/Excel. 

 

18. Edge Cases (выдержка ключевых) 

В будущие даты — нельзя (для всех). 

В прошедшие даты — можно (без ограничений). 

В день нельзя смешивать Work и Absence. 

Несколько Work Logs в день — можно (по разным проектам/задачам). 

Warning при > 12 h/день (мягкий, не блокирует). 

Absence диапазоном — выходные пропускаются, дни с Work Logs — пропускаются (сообщить). 

Перенос Absence на выходной — запрещён. 

Календарь: при смене выходной→рабочий и наличии Work Log — день становится зелёным. 

Закрытие месяца: User — readonly; Admin — может всё (в т.ч. переносить между месяцами). 

Отключение кастомного поля — скрытие с сохранением данных (reenable → восстановление видимости). 

Dropdown удаление опций — старые значения становятся «устаревшими», но сохраняются в истории.

20. User Stories (основной набор для MVP) 

Формат: Как [роль], я хочу [цель], чтобы [ценность]. AC — критерии приёмки. 

Логирование (Work): 

Как User, я хочу создать Work Log с датой, проектом, длительностью (m/h/d, комбинированно), Task Number и комментарием (обяз.), чтобы фиксировать работу. 
AC: ввод не в будущее; проект только Active; длительность конвертируется; хранение в днях; отображение в часах в отчётах. 

Как User, я хочу редактировать/удалять свои Work Logs, чтобы исправлять ошибки. 
AC: нельзя править в закрытом месяце; Admin может. 

Как User/Admin, я хочу видеть предупреждение при сумме > 12 h/день, чтобы избежать случайных ошибок. 
AC: предупреждение не блокирует сохранение. 

Отсутствия: 4. Как User, я хочу создать Vacation/Sick Leave/Day Off диапазоном, чтобы не дублировать вручную каждый день отсутствия. 
AC: создаются записи только по рабочим дням; выходные пропускаются; конфликтующие дни (с Work Log) пропускаются с уведомлением; длительность по умолчанию 1 d/день; комментарий опционален. 

Как User/Admin, я хочу редактировать/удалять любую отдельную запись, созданную из диапазона отсутствия, чтобы гибко корректировать график. 
AC: перенос на выходной — запрещён. 

Фильтры/поиск: 6. Как User, я хочу фильтровать свои записи по проекту, периоду, типу, Task Number, комментариям и кастомным полям, чтобы быстро находить нужное.
